SAP EWM User Guide

Enterprise Browser 3.4

Overview

SAP Enterprise Warehouse Manager (EWM) is an Enterprise Browser-based app for Zebra devices that can execute EWM use cases by connecting directly to an SAP S/4HANA cloud instance.

Zebra Picking Plus is a built-in solution for SAP EWM warehouse workflow use cases and requires a new license SKU to be procured. Supported only on SD660-platform devices running Android 11.

1. Introduction

2. Prerequisite

  • The user of the APP should have a valid S4HANA user name a password
  • The target Zebra devices should have active internet connection,
  • The app is installed and properly licensed
  • S4HANA cloud instance should be reachable from the target device

Zebra Picking Plus is a built-in solution for SAP EWM warehouse workflow use cases and requires a new license SKU to be procured. Supported only on SD660-platform devices running Android 11.

3. Installation

The installer is in the form of APK and it can be transferred to the device and installed by executing the apk.

The apk name is EnterpriseBrowser_SAP_[version].apk

  2. Connect the Zebra device with the laptop via cable

5. Application configuration

From the file explorer view in laptop the app folder location should look like the following.

Click image to enlarge; ESC to exit.

Hit the ‘Return to OS’ button to exit from the App.

There is a configuration file “config.xml” available inside the folder. By using the laptop file explorer move to the location where the App is installed.

Click image to enlarge; ESC to exit.

Copy the Config.xml file from device to laptop

Open the Config.xml file in any text editor

Add / Modify the following values

  1. Under <Configuration><Navigation> tag set the values of ‘Cookies’ and ‘ThirdPartyCookies’ as the following:

        
    <Navigation>
    <Cookies value="1"/>
    <ThirdPartyCookies value="1"/>
    </Navigation>

  2. Under <Configuration><Screen> tag set the values of ‘EnableZoom’ as the following

        
    <Screen>
    <EnableZoom value="0"/>
    </Screen>

  3. Under <Configuration> add the following tags

        
    <SapEwm>
    <SapUrl value="{sap url} "/>
    <Warehouse value="{Warehouse}"/>
    <Resource value = "{Resource}"/>
    <Plant value = "{Plant}"/>
    </SapEwm>

Once the values are provided configuration should look something like the following

    
    <SapEwm>
        <SapUrl value="https://my302 438 - api.s4hana.ondemand.com"/>
        <Warehouse value="Z254"/>
        <Resource value = "MOBILE 2"/>
        <Plant value = "1710"/>
    </SapEwm>

  1. Under <Applications><Applications> <IntentReceiver> tag add ‘DWProfileName’ tag and provide values of intents. This is set to receive scan data from DataWedge ( Another APP resides in the device) through intent. In normal condition (fresh installation )the values provided in the following example should be used

    
<IntentReceiver>
<EnableReceiver value="1"/>
<IntentAction value="com.zebra.action.DEFAULT"/>
<IntentCategory value="com.zebra.category.MAIN"/>
<DWProfileName value="ewmProfile"/>
</IntentReceiver>

  2. Under <Applications><Applications><General> tag set value of ‘StartPage’ as the following file

    
<General>
<StartPage value="file:///run_sap_ewm_app" name="Menu"/>
</General>

  3. Under Under <Applications<>Applications<>SIP< tag set the ResizeOnButtonbar value as following

    
<SIP>
<ResizeOnButtonbar value="1"/>
</SIP>

Save the Config.xml and transfer back to the device

Start the app again

On successful configuration following screen will appear.

Click image to enlarge; ESC to exit.

SAP server input box show the url of the SAP server provided in the Config.xml. If required end user may change it. The modified value will be updated in the Config.xml file. Once modified the app will continue use the modified url.

6. Login

Once login the app automatically logs in to the Resource mentioned in the configuration.

The following screen should appear with all the menus enabled

Click image to enlarge; ESC to exit.

If login to resource is not successful due to wrong value in configuration file or due to the resource is already occupied then the menu ‘WM Workflows’ will be in disable state. User need to go to ‘App configuration’ to provide the correct values.

If configuration does not have any Plant value then the menu ‘IM Workflows’ will also be disabled.

7. Edit configuration

User can change the Warehouse, Resource and Plant values if required.

To modify those values go to ‘App Configuration’ option.

Following screen will appear.

Click image to enlarge; ESC to exit.

Provide values and choose confirm. The app first logs off from current resource the logon to new resource.

On success it will enable the menus to execute use cases.

8. Use case execution

This application implements 13 use cases in total, 8 of them are from WM workflows and 5 of them are from IM workflows

Following picture shows the navigations from main screen to execute different use cases

Click image to enlarge; ESC to exit.

Most of the inputs are expected as scan input through device scanner. There are some inputs like quantity where keyboard would popup automatically. However for all scan input there will option to provide input through keyboard too.

8.1. WM Workflows

➢ Bin to Bin Planned

Click image to enlarge; ESC to exit.

Task created in Server beforehand. End user need to confirm the task

Flow is the like the following

  1. End user select Bin to Bin planned
  2. App queries the server to fetch the list of tasks created for Bin to Bin planned
  3. On received data from server, the is shown to the end user
  4. User choose a task
  5. App shows following required input and corresponding values expected from the end user
  6. Product number, Source Bin, Batch, Quantity, Destination Bin
  7. User provides values one by one and App matches the values provided. If there is any mismatch App shows error and ask provide the value again. If all the value matches it asks for the confirmation.
  8. Once user confirms the App sends information to the server as this task is confirmed by end user

➢ Bin to Bin Unplanned

Click image to enlarge; ESC to exit.

End user initiate the process and sends information to the server.

Flows is similar like Bin to Bin planned, only difference is that as the task is not created App does not have any expected values. App asks for input only.

The flow is like the following

  1. End user selects Bin to Bin unplanned
  2. App shows following required input expected from the end user Product number, Source Bin, Batch, Quantity, Destination Bin
  3. Once all data received app sends information to server

➢ Outbound Planned

Click image to enlarge; ESC to exit.

Delivery Item created and planned in Server and end user needs to execute the task

Flow is like the following

  1. End user selects Outbound planned
  2. App queries the server to fetch the list of tasks created for Outbound Planned
  3. On receipt App shows the data to user to select a task from there
  4. The tasks has Product Id, Source bin and Quantity predefined
  5. User choose a task App asks for inputs sequentially and matches with the value set by the server
  6. Once all input received App sends confirmation information to server

➢ Outbound Unplanned

Click image to enlarge; ESC to exit.

Delivery Item created in Server but not planned and end user needs to execute the task

Flow is like the following

  1. End user selects Outbound planned

  2. App queries the server to fetch the list of tasks created for Outbound Planned

  3. On receipt App shows the data to user to select a task from there

  4. When user choose a task App asks for following inputs sequentially

  5. Product Id, Source bin and Quantity

  6. Once all input received App sends confirmation information to server

➢ Inbound Receiving

Click image to enlarge; ESC to exit.

Delivery Item created in Server but not planned and end user needs to execute the task

Flow is like the following

  1. End user selects Inbound receiving
  2. App queries the server to fetch the list of tasks created for Inbound receiving
  3. On receipt App shows the data to user to select a task from there
  4. When user choose a task App asks for following inputs sequentially and matches with the values set by the server
  5. Product Id, Source bin, Quantity and Destination Bin
  6. Once all input received App sends confirmation information to server

➢ Cycle Count

Click image to enlarge; ESC to exit.

Shows cycle count based on Physical Inventory number

Flow is like to following

  1. App asks for PI numbers
  2. App shows list based on the input
  3. App asks for Product number, Source Bin, Quantity
  4. Once all input received user send confirmation information to server

➢ Cycle Count List

Get a list of Cycle Count from Warehouse and then execute the same flow as Cycle Count

Click image to enlarge; ESC to exit.

➢ Item Enquiry

Click image to enlarge; ESC to exit.

Show item based on user filtration criteria.

  1. App asks for the material number and queries server for the details of the material
  2. App ask user for filtration criteria source bin, batch, stock type
  3. Shows result based user selection
  4. Other than material all fields are optional. User need to tap once to select the field for scan input. Need to tap twice to enter value through keyboard.

8.2. IM Workflows

➢ Material Transfer

Click image to enlarge; ESC to exit.

In material transfer there are 3 use cases. a) 311-One Step Transfer b) 313-Remove From Storage and c) 315-Place in storage

The above picture shows steps of One step transfer

In all the cases execution flow is similar. Like the following

  1. App asks for product from user.
  2. Validate product with the server
  3. App ask for the other inputs From Location, Batch Number , Quantity, To Location and Plant
  4. Once all information received user send confirmation to server

➢ Cycle count

Click image to enlarge; ESC to exit.

The above steps shows count for one product is completed

  1. App collects of all PIs which have count pending and shows user as a list
  2. User selects one PI from the list
  3. App shows all the products under the chosen PI also shows the count status whether it is counted or not
  4. User selects on product to provide counting
  5. App asks for Product Number, Batch and Quantity
  6. Once all the information received app send information to server with the count value

Click image to enlarge; ESC to exit.

Once confirmed the corresponding product will show the completed status

➢ Stock Overview

Click image to enlarge; ESC to exit.

  1. App takes the Material Number as a mandatory input and Warehouse, Source Bin, Stock Type and Batch as optional input.
  2. Once user searches App queries server with the values provided by the user
  3. APP shows details of the Material Number such as Warehouse, Source Bin, Stock Type and Batch depending upon the input given.
